- 如果按功能层级判断，当前已经完成 `P1 核心功能` 的主链闭环，并进入 `P1 收口 / P2 增强功能` 交界状态：管理后台、工作流编辑器、用户画像、安全中心、协作视图这些可视化能力已经基本跑通，底层的 Master Bot、搜索/写作 Agent 执行链、记忆存储、基础权限与 NATS/worker 调度链也已补齐可运行闭环；后续重点转向多平台深接、外部工具执行、更强安全运营和更完整多实例治理。

## 架构理解补充

- WorkBot 的真正主链路不是“页面 -> 接口 -> 数据”，而是“接入层 Adapter -> 安全网关 -> Master Bot / Dispatcher -> Agent / 工作流 -> 存储层 -> 可视化面板”。
- 安全网关不是一个安全页面，而是进入 Master Bot 之前的可信输入边界，必须先过 5 层：限流、认证与权限、Prompt Injection 检测、内容策略处理、审计与遥测。
- `security_gateway_pipeline.svg` 明确了第 3 层需要 `正则 + 小模型分类器` 双重检测，第 4 层是“改写后放行”的通过层，不是简单硬拒，第 5 层必须是 append-only 审计日志，并且最好带 OpenTelemetry tracing。
- `memory_distillation_lifecycle.svg` 明确了记忆不是单库直存，而是三层结构：Short-term 用 Redis 滑动窗口缓存最近 20 轮、TTL 24h；Mid-term 用 SQLite 保存会话摘要；Long-term 用加密后的 ChromaDB 保存用户画像、偏好和可检索长期记忆。
- 记忆蒸馏也不是简单压缩聊天记录，而是 hierarchical summarization：先抽实体、决策、偏好、任务结果，再抽事件，最后形成高信息密度摘要；对话原始日志和记忆摘要必须分开存。
- 连续消息处理是核心能力，不是小优化。Master Bot 前面需要消息缓冲和 `debounce 2-3 秒`，还要区分“追加上下文”还是“新任务”，并支持对运行中的 Agent 下发 `context_patch`。
- 工作流编辑器真正要落地的不只是拖拽画布，而是节点绑定 `agent_id`、支持消息/定时/Webhook/内部触发，并把节点状态 `pending/running/done/error` 实时广播给前端。
- 平台接入必须按 Adapter 模式做插件化，统一接口至少包括 `receive_message`、`send_message`、`get_user_info`，这样新增钉钉、企微、TG、飞书时不会改核心逻辑。
- Agent 配置不能只停留在单一提示词文件，后续要逐步补齐 `agent.md`、`soul.md`、`tools.yaml`、`memory_rules.md` 和 `examples/` 这套配置结构。
- 语言支持已经在补充文档里明确：消息进入 Master Bot 前要做 `detected_lang` 检测，前后端后续都要为中英文双语链路留口。
