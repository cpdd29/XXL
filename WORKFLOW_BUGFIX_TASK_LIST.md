# 工作流 Bug 修复任务清单

更新时间：2026-04-22

## 基本原则

- 工作流页面配置的流程就是系统主流程。
- 一切执行都以可视化工作流为唯一真相，不允许暗路由、暗分支、暗调度。
- 画布展示什么，运行时就执行什么；运行时执行了什么，监控与画布就必须如实展示什么。
- 子工作流、Agent、回传分支都必须通过可视化节点显式表达，不能依赖隐藏逻辑兜底。

## B1 运行状态与画布状态不一致

问题：

- 当前监控/画布状态不可信。
- 主运行状态和实际执行结果不一致。
- 任务已经完成后，当前阶段仍可能停留在前置节点。

根因范围：

- 运行快照由 `graph_state.node_states + steps` 二次拼装，缺少终态统一真源。
- `current_stage` 计算优先取 `running` 节点，覆盖了真实终态。
- 当前高亮边和活跃节点存在推断逻辑，不完全来自真实执行链路。

影响文件：

- `backend/app/services/workflow_execution_service.py`
- `reception` 侧工作流监控/画布状态消费模块

修复任务：

- [ ] 建立 `run terminal state` 单一真源，禁止监控层自行猜测完成态。
- [ ] 在渲染节点状态前增加终态对账，完成态优先级必须高于历史 `running` 标记。
- [ ] 拆分 `runtime_stage`、`final_stage`、`last_completed_node`，避免一个字段同时承担运行态和结果态。
- [ ] 画布活跃边只基于真实执行过的边生成，不再使用推断式高亮。
- [ ] 监控接口与画布接口统一消费同一份运行快照版本，避免前后口径不一致。
- [ ] 明确失败、取消、完成三类终态的节点收口规则，并写入执行快照。

验收标准：

- [ ] 工作流完成后，`current_stage` 不再停留在已结束节点。
- [ ] 画布高亮节点、监控状态、最终执行结果三者一致。
- [ ] 同一次运行在刷新页面、重新拉取详情、打开监控面板时显示一致。
- [ ] 失败场景下能准确定位失败节点，而不是停留在上一个 `running` 节点。

建议回归：

- [ ] 单链路成功执行回归
- [ ] 中途失败回归
- [ ] 子工作流返回后主流程完成回归
- [ ] 人工取消后状态收口回归

## B2 重复调度与重复收口

问题：

- 同一个可视化工作流节点可能重复创建子运行。
- 子工作流已经在跑，父流程重入后仍可能再次创建和调度。
- 主流程和子流程之间存在重复收口、重复推进风险。

根因范围：

- `workflow` 节点重入时没有稳定的节点执行实例标识。
- 子工作流创建后立即自动调度，父流程又可能手动继续推进。
- dispatcher / scheduler 对同一运行中的节点缺少足够强的幂等屏障。

影响文件：

- `backend/app/services/workflow_execution_service.py`
- `backend/app/services/workflow_dispatcher_service.py`
- `backend/app/services/workflow_scheduler_service.py`

修复任务：

- [ ] 为工作流节点引入稳定的执行实例键，至少覆盖 `parent_run_id + node_id + attempt`。
- [ ] 同一节点执行实例只允许创建一个 child run，重入时必须复用已有 child run。
- [ ] `sub_workflow` 创建子运行时补齐“创建但不重复调度”或等价幂等调度机制。
- [ ] 父流程在等待子流程期间，只允许查询、轮询、收口，不允许重复创建。
- [ ] dispatcher / scheduler 增加同 run、同 node、同 attempt 的去重保护。
- [ ] 子流程完成后的收口动作只能执行一次，重复事件必须被安全丢弃。
- [ ] 将重试语义显式化，只有画布上可见的重试/再次执行动作才允许产生新 attempt。

验收标准：

- [ ] 同一可视化节点一次执行只产生一个 child run。
- [ ] 父流程重入不会再次创建相同子流程。
- [ ] 同一子流程完成事件不会导致父流程重复收口。
- [ ] 若发生真实重试，监控中可以看到明确的 attempt 递增记录。

建议回归：

- [ ] `sub_workflow` 正常等待回归
- [ ] 父流程重入去重回归
- [ ] 子流程完成事件重复投递回归
- [ ] 调度器重复 tick 幂等回归

## B3 回传分支语义未按工作流配置生效

问题：

- 配置了回传分支，但实际执行没有严格按画布语义走。
- 子工作流输出的 `handoffTarget`、`conversationStage` 等结果没有成为父流程显式分支事实。
- 运行时仍可能被 `workflow_mode`、`routeDefault`、`routeIntents` 等旧式字段隐式带路。

根因范围：

- 子工作流输出没有被系统化映射为父流程可消费的标准事实。
- `sub_workflow` 节点缺少明确的输入映射、输出映射、回传载荷定义。
- 条件判断优先级不清晰，存在旧字段覆盖可视化配置的问题。

影响文件：

- `backend/app/services/workflow_execution_service.py`
- 工作流定义与节点配置相关服务

修复任务：

- [ ] 为子工作流返回结果定义统一回传载荷，至少覆盖 `return_payload`、`handoff_target`、`conversation_stage`、`result_status`。
- [ ] 为 `sub_workflow` 节点补齐显式输入映射与输出映射配置。
- [ ] 子工作流返回后，父流程只能依据可视化条件节点或可视化边条件继续路由。
- [ ] 梳理条件判断优先级：显式子流输出 > 节点输入映射结果 > 兜底事实。
- [ ] `routeDefault` 只允许作为画布上可见的保底分支，不允许继续承担隐藏业务路由。
- [ ] 执行日志中记录“命中了哪条返回分支、依据哪个返回字段命中”。

验收标准：

- [ ] 修改画布上的回传分支配置后，不改代码也能改变运行走向。
- [ ] 子工作流返回不同 `handoff_target` 时，父流程走向不同分支。
- [ ] 运行日志能明确看到分支命中依据。
- [ ] 不再存在“画布看起来这样配，但实际被隐藏字段改道”的情况。

建议回归：

- [ ] 专业工作流返回专业分支回归
- [ ] 自由工作流返回自由分支回归
- [ ] 对话 Agent 回传到指定节点回归
- [ ] 默认分支兜底回归

## B4 联通回归与验收

目标：

- 用真实可视化主链验证“配置即执行”，确认没有旧逻辑残留继续接管流程。

回归任务：

- [ ] 基础工作流主链从入口到出口完整跑通一次
- [ ] 安全 Agent、对话 Agent、需求分发 Agent、万事通 Agent、专业工作流、自由工作流之间的可视化跳转全部留痕
- [ ] 校验画布展示节点顺序与实际执行顺序一致
- [ ] 校验子工作流返回后的分支命中结果一致
- [ ] 校验监控页面、运行详情、审计记录三方口径一致
- [ ] 校验不存在任何绕过工作流配置的旧路由、旧自动分发、旧自动补调度

完成标准：

- [ ] 以 `基础工作流 · v2.0` 为主链时，系统可以严格按画布定义执行
- [ ] 监控状态可信，运行结果可信，分支语义可信
- [ ] 重复调度、重复收口、隐式分支三类问题全部有稳定回归样本
