WorkBot 八爪鱼架构改造方案
版本：v1.0
日期：2026-04-10
目标：将当前系统进化为“大脑封闭、触手外置、协议统一”的多Agent协作平台

一、改造背景：为什么必须走向“八爪鱼”
你的项目已经从概念验证走到了可运行的MVP系统——具备了接入层、安全网关、Master Bot、工作流引擎、三层记忆、NATS通信和工具库页面。但在当前架构下，存在三个结构性风险，阻碍系统进化为真正的“数字员工团队”：

1. 大脑与触手耦合，系统臃肿且脆弱
现状：大量“重资源Skill”（PDF解析、通用搜索、未来可能的浏览器自动化）以Python函数形式内置在主服务中。

后果：

臃肿：主服务依赖pypdf、playwright等重型库，内存占用高，启动慢。

脆弱：一个Skill的崩溃（如PDF解析器遇到损坏文件）可能导致整个主进程阻塞或重启。

更新困难：修改一个Skill需要重新部署整个大脑。

2. 专业工作流缺少安全的执行边界
现状：专业工作流（professional_workflow_service）尚未真正落地，权限型任务（如查询CRM、操作订单）的代码如果也写在大脑内部，会将敏感凭证和业务逻辑暴露在核心服务中。

需求：你设想的“像人一样处理需求”的专业工作流，必须运行在独立、有权限隔离的环境中。

3. 大脑的“指挥官”角色被淹没在日常执行中
现状：Master Bot既要负责意图识别和路由，又要执行部分本地工具调用。

理想：大脑应该只做决策（分析意图、编排工作流、选择触手），而所有执行都交给外置的Agent或工具。

八爪鱼架构的核心主张：

大脑封闭、触手外置、协议统一（MCP）。
让核心系统成为轻量、稳定、高智商的中枢，让外部服务成为可插拔、可独立演进的执行单元。

三、多Agent开发方式：分层协作模型
在八爪鱼架构下，多Agent开发遵循 “大脑编排，触手执行” 的模式。不同Agent的物理部署位置不同，但通过NATS/MCP统一通信。

1. Agent的分类与边界
类型	部署位置	职责	通信方式	示例
大脑型Agent	主服务内部（Python类）	需求澄清、意图识别、工作流编排、结果聚合。不执行具体工具操作。	内部函数调用	项目经理Agent(PM Agent)、工作流路由器
工具型触手Agent	独立MCP服务	执行单一领域任务，如搜索、PDF解析、画图。可以是纯工具，也可以带有LLM推理。	MCP协议(stdio/sse/http)	搜索触手、PDF处理触手、浏览器触手
专业型触手Agent	独立微服务（MCP或HTTP）	处理需要特定权限、私有知识库或复杂业务逻辑的任务。内部可包含多步推理。	MCP或HTTP API	合规审查专家、CRM订单查询助理、财报分析员
2. 协作流程（以一次“帮我查一下广州天气并写一份出行建议”为例）
用户消息 → 钉钉 → 接入层Adapter → 统一消息对象。

安全网关：通过五层过滤。

项目经理Agent(大脑)：

分析意图：weather_query + text_generation。

判断为自由工作流，无需权限。

创建任务，生成执行计划：

Step 1: 调用 weather-mcp 触手获取天气。

Step 2: 调用 writer-agent 触手生成建议。

工作流引擎(大脑)：

通过 MCP Runtime 调用 weather-mcp 的 get_weather 工具，等待返回。

将天气数据作为上下文，通过 MCP Runtime 调用 writer-agent 的 generate_text 工具。

结果聚合：大脑将触手返回的文本稍作润色，通过 channel_outbound_service 发回钉钉。

记忆蒸馏：夜间，记忆系统从对话中提取“用户关注天气”，存入长期画像。

四、模型角色分配：GPT-5.4 vs GPT-5.3 Codex 如何分工
你拥有两个能力侧重点不同的模型，应该让它们各司其职，实现智力分级，最大化性价比。

1. 模型能力画像
模型	核心优势	弱点	适用场景
GPT-5.4	强推理、强规划、深度理解。擅长处理模糊指令、多步推理、策略制定、结果评估。	相对较慢、成本较高。	意图识别、工作流编排、复杂问题澄清、最终结果审核、记忆蒸馏。
GPT-5.3 Codex	代码生成能力极强、工具调用精准、响应快、成本低。擅长结构化输出、遵循JSON Schema、执行明确指令。	在处理极度开放的闲聊或哲学问题时不如5.4细腻。	执行具体工具调用、解析数据、生成代码、填充模板、翻译、总结。
2. Agent与模型的绑定策略
大脑层 (必须使用GPT-5.4)

项目经理Agent(PM Agent)：需要从模糊对话中澄清需求，必须用GPT-5.4。

工作流路由器：判断走自由/专业工作流，涉及权限评估，用GPT-5.4。

记忆蒸馏器(夜间任务)：从大量对话中提取结构化偏好和决策，需要强归纳能力，用GPT-5.4。

触手层 (可配置，推荐GPT-5.3 Codex)

搜索触手：接收明确的搜索词，返回结构化结果，用GPT-5.3 Codex足够。

写作触手：根据大纲生成文本，属于明确指令，可用GPT-5.3 Codex（但若要求“文案富有诗意”，则建议用5.4）。

PDF解析触手：无LLM参与，纯解析工具。

代码生成触手：这必须是GPT-5.3 Codex的本职工作。

专业Agent(如CRM查询)：将自然语言转SQL或API调用，GPT-5.3 Codex极其擅长。

3. 模型降级与成本控制
在系统配置(agents/*/agent.md)中为每个Agent设置模型偏好：

yaml
# agents/pm_agent/agent.md
model:
  primary: "gpt-5.4"
  fallback: "gpt-5.3-codex"   # 当5.4不可用或超时时降级
  cost_optimization: 
    enable: true
    simple_intent_use: "gpt-5.3-codex"  # 对于"你好""谢谢"等简单意图直接用便宜模型
4. 推荐的Agent数量与分工
阶段一（当前至2周内）：保持 2-3个大脑Agent + 3-5个触手。

大脑：项目经理、路由器。

触手：搜索、PDF、写作、天气、任务查询。

阶段二（1个月后）：增加 1-2个专业触手。

例如：CRM查询触手、内部文档RAG触手。

阶段三（长期）：根据业务需求动态增加，但大脑Agent数量保持稳定（不超过5个），触手可以无限扩展。每个触手都是一个独立的MCP服务，而不是主项目里的一个类。

附录：与现有文档的关系
本清单与 addSkill.md 中的 P3 MCP 整合任务高度一致，应将其优先级提升。

本清单中的专业工作流部分对应 tode.md 中 P2-2 和 P2-3 任务，提供了更清晰的实施路径。

模型分工部分可作为 agents/*/agent.md 配置的指导原则。

开发SOP可作为 开发指南补充.md 的后续章节。

改造的核心思想是渐进式重构，不推翻现有系统，而是通过“外科手术”将臃肿部分剥离，最终进化成一个清晰、健壮、可无限扩展的八爪鱼架构。
