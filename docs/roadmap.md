# WorkBot Roadmap

版本：v1.1  
更新时间：2026-04-11  
状态：执行版路线图

## 1. 项目背景

WorkBot 当前已经不是纯原型，而是一个可运行的多 Agent MVP：

- 已有统一消息入口、平台 Adapter、安全网关、任务中心、工作流引擎、三层记忆、NATS 基础链路
- 已有基础 Agent 执行链、自定义工作流、工具库页面、能力来源扫描与 MCP bridge 雏形
- 已具备 `chat / free_workflow / professional_workflow` 的初步分层能力

但当前项目仍存在三个核心问题：

1. 核心服务承担了过多执行职责，容易越做越重
2. 自由工作流、专业工作流、外部能力桥接还没有完全按统一治理方式收口
3. 企业可落地所需的权限、审计、审批、定时任务、只读/写操作边界还不够清晰

本路线图的目标，不是继续无序堆功能，而是把项目拆成清晰的几块，按阶段稳定推进，最终演进成：

- 一个稳定、轻量、统一的“大脑”
- 一组可插拔、可桥接、可替换的“触手”
- 一套能进入企业场景的专业工作流与安全治理体系

---

## 2. 最终目标

WorkBot 要实现的不是“万能闲聊机器人”，而是“能接待、能判断、能执行、能追踪、能记忆、能定时”的工作型系统。

目标体验分为三类：

### 2.1 专业工作流

典型目标：

- 用户提出业务事项，例如“xx 客户下了 200 个鼠标垫的订单”
- 系统识别为专业工作流
- 系统发起确认，而不是直接误执行
- 用户确认后创建任务并开始流程
- 每经过一个流程节点都通知用户
- 用户可以随时查询任务进度

### 2.2 自由工作流

典型目标：

- 天气、搜索、写作、任务查询、文档处理等问题直接进入自由工作流
- 系统尽量少确认、少兜圈，直接把结果做出来
- 出错时能降级，而不是退回空泛接待话术

### 2.3 定时与长期协助

典型目标：

- 用户用自然语言创建周期任务，例如“每周五下午三点发一份周报给我”
- 系统把它转成正式调度规则
- 到点自动执行并回传结果

## 3. 架构原则

### 3.1 总原则

1. 先做“统一大脑”
2. 再做“稳定自由工作流”
3. 再长“第一个触手”
4. 再做“只读专业工作流”
5. 最后才做“审批和写操作”

### 3.2 八爪鱼原则

1. 大脑封闭
2. 触手外置
3. 接入统一
4. 配置统一
5. 安全前置
6. 先查后改

### 3.3 一条硬规则

大脑只负责：

- 接待
- 意图识别
- 路由分流
- 工作流编排
- 能力选择
- 结果聚合
- 状态追踪

大脑不负责：

- 直接执行重资源能力
- 直接持有专业系统业务逻辑
- 直接内嵌高风险写操作

## 4. 项目拆分边界

## 4.1 保留在主系统的大脑部分

这些能力必须留在主项目内，作为稳定中枢：

- 接入层 Adapter
- 安全网关
- Master Bot / Router
- 工作流编排引擎
- 任务中心
- 任务状态与进度通知
- 三层记忆
- Tool Gateway / MCP Runtime
- 工具库与来源管理
- 审计与权限治理
- 定时任务调度器
- 配置真源与能力注册中心

## 4.2 需要逐步桥接出去的触手部分

这些能力不应长期内嵌在主服务里：

- PDF 处理
- PDF 转 Word
- 通用搜索
- 浏览器自动化
- 专业系统查询
- 内部知识库外部 Agent
- 行业型专项触手

## 4.3 当前保留在主脑内的轻能力

当前实际保留在主脑内的，仅允许是只读轻能力与编排视图能力：

- 任务状态查询
- 任务列表查询
- 基础结果聚合与说明生成

说明：

- `weather_skill`
- `speech_writer_skill`
- `general_writer_skill`

以上三类已不再保留在主仓执行链中，默认必须走外部 runtime。

## 4.4 当前明确不优先做的内容

- 同时接入多个专业系统
- 高风险写操作
- 财务/审批直接执行
- 重浏览器自动化
- 一次性删除所有内置 Skill
- 复杂多租户治理
- 在线编辑所有工具配置

## 4.5 当前能力去向清单

下面这份清单用于明确“哪些先保留在大脑里，哪些需要桥接出去，哪些暂缓”。

### 当前先保留在主系统内

- `task_status_skill`
- `task_list_skill`
- `professional_workflow_service`
- `master_bot_service`
- `agent_execution_service`
- `tool_catalog_service`
- `tool_source_service`
- `mcp_runtime_service`

保留原因：

- 这些能力当前更偏编排、只读查询、结果聚合或治理中枢
- 先保留有利于稳定主链，避免拆太快导致整体不可控

### 第一批已桥接出去

- `pdf_read_skill`
- `pdf_summary_skill`
- 新增 `pdf_to_docx`

桥接目标：

- 建立第一个独立 MCP 触手
- 打通双跑、灰度、切流、回滚这整套迁移模式

### 第二批已桥接出去

- `web_search_skill`
- `weather_skill`
- `speech_writer_skill`
- `general_writer_skill`

桥接目标：

- 验证“第二个触手可复制”
- 让大脑不再承担搜索、天气、写作类重执行职责

### 第三批再桥接出去

- 浏览器自动化
- 内部知识库专项检索 Agent
- 行业专项 Agent

前提条件：

- 第一批和第二批触手模式稳定
- MCP Runtime 已具备健康检查、降级、回滚、统计

### 只读专业触手

- 订单查询触手
- CRM 客户信息查询触手
- 报表导出触手

要求：

- 先做只读
- 先有权限边界
- 先有审计

### 暂缓到最后

- 订单创建
- 订单修改
- 财务审批
- 高风险业务写操作
- 自动化批量执行

## 5. 开发方式与执行规则

## 5.1 配置真源

必须统一一个配置真源，避免配置漂移。

建议：

- 运行态真源：数据库中的 source registry / tool registry
- 文件态：只作为开发导入模板，不作为最终运行真源

禁止长期并存的混乱模式：

- 一部分写在 `yaml`
- 一部分写在数据库
- 一部分靠代码硬编码

## 5.2 外置触手迁移策略

任何内置 Skill 外置时，必须走四步迁移：

1. 双跑
2. 灰度
3. 切流
4. 删除

详细要求：

- 双跑阶段：内置版和 MCP 版同时保留
- 灰度阶段：只让部分请求走 MCP
- 切流阶段：默认走 MCP，但保留回滚开关
- 删除阶段：确认稳定后才删内置版

禁止直接做的事：

- MCP 一接通就删内置实现
- 没有健康检查就强制切流
- 没有回滚开关就全量上线

## 5.3 专业工作流上线顺序

专业工作流必须按以下顺序做：

1. 先做准入判断
2. 先做只读流程
3. 再做人工确认
4. 最后做写操作

换句话说：

- 先做“查”
- 再做“批”
- 最后做“改”

## 5.4 安全开发基线

所有专业触手、审批节点、写操作流程必须自带以下字段或机制：

- `audit_id`
- `idempotency_key`
- `execution_scope`
- `approval_required`
- 短时效服务间凭证
- 超时 / 重试 / 熔断
- 失败补偿或人工接管

## 5.5 每项任务都要满足的开发闭环

1. 设计输入输出 Schema
2. 定义调用边界
3. 完成实现
4. 补齐测试
5. 接入工具库可见性
6. 补齐日志与健康检查
7. 写明回滚方式

## 阶段 1：统一大脑

目标：把大脑做轻、做稳、做统一。

### 任务清单

- [x] R1-1 固化 `chat / free_workflow / professional_workflow` 三段式路由
- [x] R1-2 为每次路由产出统一的 `route_decision`
- [x] R1-3 固化专业流程确认节点
- [x] R1-4 固化任务中心：任务创建、节点状态、执行 trace、用户进度查询
- [x] R1-5 把自由/专业工作流都接入统一任务模型
- [x] R1-6 把“接待、判断、执行、回传”日志串成同一条可追踪链

### 关键结果

- 用户发一句话，大脑先判断，不直接盲做
- 专业流程会先确认
- 用户可随时追问“现在进度怎么样”

### 验收标准

- 场景“客户下单了 200 个鼠标垫”能先触发专业流程确认
- 确认后任务进入正式流程
- 每个节点有状态、时间、说明、责任模块

  目标：出问题时立刻切回内置。  
  完成标准：单开关即可回退到内置实现，不需要改代码。

  前提：双跑稳定、灰度稳定、回滚验证完成。

- [x] D3-09 补齐 PDF 触手端到端测试
