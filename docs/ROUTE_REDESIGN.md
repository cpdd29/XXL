# 当前可用功能与路由重整建议

更新时间：2026-04-16

当前状态：

- 第一轮信息架构已经在前端落地：侧边栏已改成 `工作台 / 风险中心 / 能力接入 / 系统设置` 四组，用户可见名称已切到“总览 / 待办中心 / 任务中心 / 执行过程 / 能力中心 / 模型与 Provider / 配置来源与优先级 / 外接治理”这套口径。
- `总览` 页已完成第一轮重构：当前结构调整为“状态摘要 + 当前待办 + 风险提醒 + 分页签查看运行细节”，不再把调度、成本、能力、日志全部堆在同一屏长页面里。
- 旧 URL 暂未迁移，当前仍以兼容路径运行；本文件中的 `/home / inbox / jobs / risk / connections` 等路径属于下一轮 URL 重构建议。

## 1. 这份文档的目标

当前前端页面已经不少，但问题不是“功能不够”，而是：

- 路由命名偏开发视角
- 顶层导航把低频后台页和高频工作页混在一起
- 同一条业务链被拆到多个并列入口，用户不知道该先去哪里
- 一些页面更适合作为“详情子页”或“设置子页”，不适合放顶层

这份文档先回答两个问题：

1. 当前项目到底已经能用哪些功能
2. 这些功能应该如何按“用户任务流”重组路由

---

## 2. 当前项目已经可用的功能点

以下只统计当前仓库中已经有前端页面和实际数据支撑的能力，不把“未来设想”算进去。

### 2.1 日常运营 / 一线工作

- 控制台
  - 查看 SLA、运行指标、调度状态、告警准备、失败归因、待回复队列、经理待办
  - 当前路由：`/dashboard`
- 接待工作台
  - 按渠道查看待回复会话
  - 回到具体任务、协作处理和工作流落点
  - 当前路由：`/reception`
- 任务列表
  - 查看全部任务的状态、优先级、失败阶段、路由策略、负责人、回传状态
  - 支持搜索、筛选、取消、重试
  - 当前路由：`/tasks`
- 任务详情
  - 查看执行结果、执行步骤、任务摘要、路由决策、主脑分发摘要
  - 已经能承载“单个任务的完整复盘”
  - 当前路由：`/tasks/[taskId]`

### 2.2 编排与执行可视化

- Agent 协作可视化
  - 查看任务流转、节点状态、执行日志、项目经理分发、主脑分发摘要、路由决策、回退历史
  - 当前路由：`/collaboration`
- 工作流编辑器
  - 查看和编辑工作流定义
  - 当前路由：`/workflow`

### 2.3 安全与风险治理

- 安全中心
  - 审计日志
  - 安全规则
  - 策略配置
  - 处罚运营
  - 安全报告
  - 当前路由：`/security`
- 统一告警中心
  - 查看安全告警与筛选
  - 当前路由：`/security/alerts`
- 审批中心
  - 提交和处理高风险配置、安全放行、人工接管等审批单
  - 当前路由：`/settings/approvals`

### 2.4 外接能力与渠道接入

- 工具库
  - 查看能力列表、外部来源、桥接治理状态、来源健康
  - 当前路由：`/tools`
- 外接能力治理
  - 按 family 治理外接 Agent / Skill
  - 支持切主、fallback、灰度、回滚、弃用、查看治理审计
  - 当前路由：`/settings/external-capabilities`
- 渠道接入配置
  - Telegram / 钉钉 / 企微 / 飞书等接入配置
  - 当前路由：`/settings/channel-integration`

### 2.5 组织与主体管理

- 用户管理
  - 查看用户列表、角色、状态、渠道信息
  - 当前路由：`/users`
- 用户详情 / 用户画像
  - 编辑画像、查看活动轨迹、账户摘要、推荐动作
  - 当前路由：`/users/[userId]`
- Agent 管理
  - 查看 Agent 列表、状态与任务完成情况
  - 当前路由：`/agents`

### 2.6 系统设置与运维控制面

- 通用设置
  - 界面与刷新行为
  - 当前路由：`/settings/general`
- 模型 / Provider 配置
  - OpenAI / Codex / Claude / Kimi / Gemini / DeepSeek / OpenAPI compatible 等配置
  - 当前路由：`/settings/agent-api`
  - 兼容别名路由：`/settings/api`
- 配置治理
  - 查看配置分层、生效来源、优先级、风险提示、配置审计
  - 当前路由：`/settings/governance`

## 3. 现有路由的主要问题

### 3.1 顶层导航按“技术模块”切，不按“用户任务”切

当前侧边栏是：

- 控制台
- 接待工作台
- 工作流编辑器
- Agent 协作
- 任务管理
- Agent 管理
- 工具库
- 用户管理
- 安全中心
- 设置

问题在于：

- 普通运营用户并不关心“协作”和“工作流编辑器”为什么是顶层并列
- “工具库”和“外接能力治理”实际属于同一域，却被拆成两个层级
- “审批中心”属于高风险治理，却被埋在设置里
- “任务详情”已经有完整复盘能力，但“协作”却单独做成顶层，导致用户分不清看任务还是看协作

### 3.2 页面命名偏内部术语

以下名称更像给研发或架构师看的：

- `Agent 协作`
- `工具库`
- `配置治理`
- `外接能力治理`
- `Agent API 配置`
- `工作流编辑器`

这些名字不是不能用，而是不能直接作为一线用户主导航名称。

### 3.3 高频页和低频页没有分层

高频：

- 控制台
- 接待工作台
- 任务
- 安全告警 / 安全日志

低频：

- 工作流编辑
- 外接能力治理
- Provider 配置
- 配置治理
- Agent 管理

现在的问题是，低频后台页占了太多顶层注意力。

### 3.4 同一业务链被拆散

一条完整的处理链其实是：

- 接待
- 路由
- 执行
- 回传
- 复盘

但现在被拆成：

- `/reception`
- `/tasks`
- `/tasks/[taskId]`
- `/collaboration`
- `/workflow`

用户很难理解这些入口之间的先后关系。

---

## 4. 建议的用户视角信息架构

建议不要再按“开发模块”组织顶层，而是改成下面四大域：

- 工作台
- 风险与治理
- 能力与接入
- 系统设置

---

## 5. 建议的新路由树

下面这版重点不是 URL 漂不漂亮，而是“用户一眼能知道去哪”。

### 5.1 一级导航

- `/home` 总览
- `/inbox` 待办 / 接待
- `/jobs` 任务中心
- `/risk` 风险中心
- `/connections` 能力接入
- `/settings` 系统设置

如果暂时不想大改 URL，至少可以先只改“菜单名称”和“分组结构”，底层仍然保留旧路由。

---

## 6. 新路由与旧路由映射建议

### 6.1 工作台域

- `总览`
  - 新建议：`/home`
  - 旧路由：`/dashboard`
  - 面向对象：管理员 / 运营 / 值班
  - 功能：系统总览、SLA、运行风险、经理待办、待回复队列

- `待办中心`
  - 新建议：`/inbox`
  - 旧路由：`/reception`
  - 面向对象：接待 / 项目经理 / 一线运营
  - 功能：待回复会话、澄清任务、按渠道处理会话

- `任务中心`
  - 新建议：`/jobs`
  - 旧路由：`/tasks`
  - 面向对象：运营 / 管理员
  - 功能：查看全部任务、筛选、取消、重试

- `任务详情`
  - 新建议：`/jobs/[taskId]`
  - 旧路由：`/tasks/[taskId]`
  - 功能：执行结果、执行步骤、路由决策、主脑分发摘要

- `执行过程`
  - 新建议：`/jobs/[taskId]/execution`
  - 旧路由：`/collaboration?taskId=...`
  - 功能：节点状态、执行日志、可视化协作流
  - 建议：不要再单独做顶层导航，而应成为任务详情的子页或二级页

### 6.2 风险与治理域

- `安全中心`
  - 新建议：`/risk/security`
  - 旧路由：`/security`
  - 功能：审计日志、规则、策略、处罚、报告

- `告警中心`
  - 新建议：`/risk/alerts`
  - 旧路由：`/security/alerts`
  - 功能：统一告警、筛选、定位风险项

- `审批中心`
  - 新建议：`/risk/approvals`
  - 旧路由：`/settings/approvals`
  - 功能：高风险动作审批
  - 建议：从“设置”里提出来，因为它本质是治理入口，不是普通设置项

### 6.3 能力与接入域

- `能力中心`
  - 新建议：`/connections/capabilities`
  - 旧路由：`/tools`
  - 功能：能力列表、来源列表、桥接状态、来源健康

- `外接治理`
  - 新建议：`/connections/governance`
  - 旧路由：`/settings/external-capabilities`
  - 功能：Agent / Skill family 的切主、fallback、灰度、回滚、弃用

- `渠道接入`
  - 新建议：`/connections/channels`
  - 旧路由：`/settings/channel-integration`
  - 功能：Telegram / 钉钉 / 企微 / 飞书接入

- `工作流设计`
  - 新建议：`/connections/workflows`
  - 旧路由：`/workflow`
  - 功能：工作流编辑
  - 建议：不要放在首页顶层，它属于配置 / 设计后台，不是一线常用页

### 6.4 系统设置域

- `通用设置`
  - 新建议：`/settings/general`
  - 旧路由：`/settings/general`

- `模型与 Provider`
  - 新建议：`/settings/providers`
  - 旧路由：`/settings/agent-api`
  - 兼容入口：`/settings/api`
  - 功能：OpenAI / Codex / Claude / Kimi / Gemini / DeepSeek 等
  - 建议：页面名称不要再叫“Agent API 配置”，用户其实在配的是模型供应商和网关

- `配置治理`
  - 新建议：`/settings/config-governance`
  - 旧路由：`/settings/governance`
  - 功能：配置来源、生效优先级、配置审计

- `用户管理`
  - 新建议：`/settings/users`
  - 旧路由：`/users`
  - 功能：用户列表、用户画像、活动轨迹

- `Agent 管理`
  - 新建议：`/settings/agents`
  - 旧路由：`/agents`
  - 功能：Agent 状态、任务完成情况
  - 建议：这是后台运维页，不建议继续放顶层

## 7. 最适合先做的第一轮收口

如果这一轮只做“最小但收益最大”的整理，我建议先做下面几件事：

### 7.1 只保留 6 个顶层入口

- 总览
- 待办中心
- 任务中心
- 风险中心
- 能力接入
- 系统设置

### 7.2 从顶层拿掉这些页

- `工作流编辑器`
- `Agent 协作`
- `Agent 管理`
- `工具库`

不是删掉，而是移动到二级导航：

- 协作页挂到任务详情下面
- 工作流编辑器挂到“能力接入”
- Agent 管理挂到“系统设置”
- 工具库并入“能力中心”

### 7.3 页面名称换成用户语言

建议替换：

- `接待工作台` -> `待办中心`
- `任务管理` -> `任务中心`
- `Agent 协作` -> `执行过程`
- `工具库` -> `能力中心`
- `外接能力治理` -> `外接治理`
- `Agent API 配置` -> `模型与 Provider`
- `配置治理` -> `配置来源与优先级`

### 7.4 能合并的页面先合并认知，不一定立刻合并代码

优先级最高的认知合并：

- `/tools` + `/settings/external-capabilities`
  - 对用户来说都属于“能力接入”
- `/tasks/[taskId]` + `/collaboration`
  - 对用户来说都属于“任务详情 / 执行过程”
- `/security` + `/security/alerts` + `/settings/approvals`
  - 对用户来说都属于“风险中心”

## 9. 哪些页面更适合“普通用户”，哪些更适合“管理员”

### 9.1 普通运营用户常用

- 总览
- 待办中心
- 任务中心
- 任务详情
- 风险中心中的告警 / 安全日志

### 9.2 管理员 / 运维 / 架构负责人常用

## 10. 结论

当前项目的核心问题不是功能少，而是：

- 页面入口按开发模块拆
- 用户任务流没有被当成主线
- 高低频页面没有分层
- 后台治理页占用了过多顶层注意力

建议下一步按下面顺序推进：

1. 先改侧边栏信息架构和菜单名称
2. 再给页面增加新的用户向别名路由
3. 最后再决定是否真正迁移旧 URL

如果要直接进入实现，第一轮最值得做的是：

- 重做侧边栏分组
- 给 `/collaboration` 降级为任务详情子入口
- 合并“工具库 / 外接治理”为同一导航域
- 把“审批中心”从设置中提到风险中心
