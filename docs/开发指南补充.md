接入层扩展：用 Adapter 模式做插件化，每个平台实现一个统一接口（receive_message / send_message / get_user_info），主系统只认标准消息对象。钉钉、企微、TG 先做，微信个人号目前只能走逆向协议（风险较高，建议用企业微信替代），飞书有完善的官方 API 可以直接接。每加一个平台只需新增一个 Adapter 类，核心逻辑零修改。
核心调度引擎：Master Bot 内部有一个专门的 Dispatcher Agent，职责是意图识别 + 任务路由。它用 LangGraph 构建，接收标准化消息后先做意图分类（调用小模型，速度快），然后决定走哪条工作流或召唤哪些子 Agent。它本身不干活，只做"大脑"角色。

安全网关的核心要点：第③层的 prompt injection 要用"正则 + 小模型"双重检测，单靠正则会漏掉语义变体，单靠模型太慢。第④层的 content policy 属于通过层而非硬拒，会对内容改写后放行，记录差异。第⑤层审计是无论通过还是拒绝都写日志，且必须是 append-only 防篡改。

现在回答7个注意事项问题。
① 流式/连续消息处理
客户连续发消息是最常见的场景（"等等"、"补充一下"、"还有..."）。解决方案是在 Master Bot 前加一个消息缓冲队列，对同一用户的消息做 debounce（比如等待 2-3 秒无新消息才触发处理）。同时引入意图合并逻辑：如果用户在 Agent 已经开始工作时继续补充，判断是"追加上下文"还是"新任务"，前者注入到当前任务的上下文里，后者进队列排队。已经在执行的 Agent 可以接受 context_patch 信号实时更新自己的任务描述。
② 记忆蒸馏时机与方法
下面是三层记忆的生命周期图：

图片：memory_distillation_lifecycle.svg

蒸馏方法推荐用 hierarchical summarization：先让 LLM 提取关键实体（人名、决策、偏好、任务结果），再提取重要事件，最后生成一段自然语言摘要。不是简单压缩，而是"提炼信息密度"。Long-term 层用向量存储后，每次新对话开始时语义检索最相关的 5-10 条记忆注入上下文，不会因历史太长导致 token 溢出。
③ 客户端与手机端接入
桌面客户端推荐用 Tauri（比 Electron 轻 10 倍，基于 Rust），连接后端的 REST API 和 WebSocket，前端代码完全复用 Web 版。手机端用 React Native 或直接做 PWA（Progressive Web App，装到手机桌面，开发成本最低）。两端都只是 UI 壳，核心逻辑全部在后端，维护成本极低。
④ 工作流绑定 Agent、监控与触发
工作流的每个节点有一个 agent_id 字段，指向具体 Agent。绑定逻辑是在工作流配置文件（workflow.yaml）里声明，节点类型有三种：agent_node（调用 Agent）、condition_node（判断分支）、tool_node（直接调用工具，不经过 Agent）。
触发方式四种：消息关键词触发、定时 cron 触发、Webhook 外部触发、Agent 内部触发（一个 Agent 完成后激活下一个工作流）。实时监控用 WebSocket 推送，每个节点状态变更（pending/running/done/error）都广播给前端，ReactFlow 收到后更新节点颜色和进度条。
⑤ 客户信息管理
每个客户对应一条 User 记录，包含：跨平台账号映射（同一人在 TG 和钉钉的 ID 关联起来）、权限等级、偏好设置、标签（可手动打标）、来源渠道、最后活跃时间。对话历史和记忆分开存，对话历史是原始日志（可查询），记忆是经过蒸馏的结构化信息。建议管理后台做一个简单的 CRM 视图，支持搜索、筛选、给用户手动备注。
⑥ Agent 配置文件不够，建议扩展为 5 个文件
/agents/search_agent/
  ├── agent.md          # 能力声明、路由规则、模型选择
  ├── soul.md           # 人格、语气、核心原则
  ├── tools.yaml        # 工具定义 + 每个工具的权限
  ├── memory_rules.md   # 该 Agent 应该记什么/忘什么
  └── examples/         # few-shot 示例（好/坏回复对）
      ├── good.md
      └── bad.md
tools.yaml 是关键补充，把工具的参数格式、调用限制、失败处理都声明清楚，方便热更新工具能力。memory_rules.md 告诉 Agent "你应该主动记住用户的 XX 偏好"，不同 Agent 记忆策略不同。examples/ 目录用 few-shot 定向纠正 Agent 的行为，比任何 prompt 工程都稳定。
⑦ 语言支持（中英文）
后端用 langdetect 或 fasttext 在消息进入 Master Bot 前做一次语言检测，结果存入对话上下文（detected_lang: zh / en）。每个 Agent 的 prompt template 准备中英文两套，根据检测结果选择。用户也可以在个人偏好里手动设置首选语言，优先级高于自动检测。前端 i18n 用 react-i18next，翻译文件放在 /locales/zh.json 和 /locales/en.json。暂时只维护这两套，后续加其他语言直接新增 json 文件，框架不用动。
