# XXL 摩天大楼像素风实施方案

## 1. 文档目的

这份文档用于替代早先那份偏目标描述的 `Pixel Skyscraper` 文本方案，给 `XXL` 项目当前的摩天大楼页面提供一份可直接执行的像素风开发路线。

这份方案只服务于 `XXL` 现有代码库，不假设新开仓库，也不假设重做后端。

## 2. 当前项目事实

- 当前摩天大楼页面的新挂载路由是 `reception/app/visualization/skyscraper/page.tsx`，旧的一级路由版页面已经移除。
- 当前前端实际技术栈是 `Next.js 16 + React 19 + Tailwind CSS 4 + React Query + next-themes`。
- 当前后端实际技术栈是 `FastAPI`，不是 `Node.js + NestJS`。
- 当前摩天大楼页已经通过 `useDashboardStats()` 读取现有 dashboard 数据，不需要为了像素风重做一套后端。
- 当前项目已经具备明亮 / 暗黑主题基础，摩天大楼页面后续可以在像素风方案里继续保留 light / dark 两套主题表现。

结论：

- 这次开发不是从零开始做一个独立 demo。
- 这次开发应该建立在 `XXL` 现有 Next 前端和 FastAPI 后端上。
- 像素风改造的重点是前端渲染层、资源层和实时表现层，不是后端重构。

## 3. 目标定义

目标不是把现在的摩天大楼页面换成一层“像素皮肤”，而是把它升级为具备游戏感的像素城市视图。

目标效果：

- 城市天际线和主楼体具有稳定的像素风细节。
- 每一栋楼都能承载运行态、健康态和告警态。
- 可以逐步加入日夜系统、灯光系统、环境动画、角色动画和状态特效。
- 页面保留现有业务数据含义，但视觉表达改成“像游戏场景，而不是网页卡片”。

非目标：

- 不在第一阶段重做整个后台接口。
- 不把所有界面元素都塞进 Pixi。
- 不继续用纯 CSS 去模拟整套游戏场景。
- 不把像素资源散落在页面组件目录里。

## 3.1 参考图结论

根目录下新增的 `bright.png` 和 `dark.png` 已经给出了非常明确的目标风格。这两张图不是简单的换色版本，而是两套完整的场景母版：

- `bright.png`：白天形态，蓝天、高反射玻璃、远景城市、屋顶设备、清晰的办公层结构、楼下街道人流车流更明显。
- `dark.png`：夜晚形态，暖色室内灯光、深色楼体轮廓、屋顶氛围灯、月亮与夜云、街道车灯、楼体发光标识和更强的层次反差。

从这两张图可以直接得出三个约束：

- 页面如果要做细致，不能只准备一套楼体贴图再通过 CSS 或滤镜硬切白天和夜晚。
- 这两张图只能作为概念母版和拆层参考，不能直接当成最终运行时主图。
- 运行时必须准备“结构层 + 光照层 + 室内层 + 环境层 + 前景层”的静态资源体系，否则做不出这种细节密度。

## 4. 为什么必须引入 PixiJS

如果目标只是做像素风按钮、边框、卡片，纯 DOM + CSS 足够。

但当前摩天大楼页面的目标已经超出普通网页像素风的上限，后续需要的能力包括：

- 统一 ticker 驱动的楼体动画
- 像素级的缩放控制
- 整数坐标与 nearest-neighbor 渲染
- 大量 sprite、灯光状态、角色状态和环境特效
- 前景 / 中景 / 背景分层与视差
- 日夜切换与运行态特效切换

继续用纯 CSS 的问题：

- 细节很容易看起来像“网页做旧”，而不是游戏场景。
- 楼体、窗户、人物和环境一多，DOM 会迅速失去统一感。
- 半像素偏移、模糊和缩放发虚很难完全避免。
- 复杂动态场景会让代码职责越来越混乱。

结论：

- `PixiJS` 负责场景渲染，是必要项。
- `React` 继续负责页面结构、交互壳、数据接入和外围控制。
- 这次应采用“React 管 UI，Pixi 管场景”的混合架构。

## 5. 建议技术方案

建议采用以下组合：

- 页面框架：`Next.js App Router`
- UI 壳层：`React + Tailwind + shadcn 风格组件`
- 场景渲染：`PixiJS`
- 服务端状态：`React Query`
- 场景运行时状态：`Zustand`
- 实时更新：复用现有 dashboard realtime / websocket 能力
- 主题切换：复用现有 `next-themes`

说明：

- `Zustand` 不用于替代现有业务接口查询，而是用于 Pixi 场景内部的运行态存储。
- `PixiJS` 尽量直接使用，不把每个 sprite 都包装成 React 组件。
- 如果后续需要 `@pixi/react`，也只应把它作为轻量桥接层，而不是场景主结构。

## 6. 推荐职责边界

### React 负责

- 页面路由、鉴权和布局壳层
- Header / Sidebar / 外围面板
- 筛选器、按钮、状态说明、错误态、空态
- dashboard 数据拉取和 websocket 生命周期管理
- 将后端数据归一化为场景可消费的数据结构
- 主题、偏好设置、调试开关

### Pixi 负责

- 主城市场景渲染
- 楼体、窗户、灯光、告警层
- 前景 / 中景 / 背景分层
- 角色 sprite 与状态切换
- 天气、蒸汽、告警灯、扫描线、光点等特效
- 日夜切换和统一 update loop
- 摄像机、视差、像素缩放和动画节奏

### 明确限制

- 不要把每一扇窗、每一个人物都建成 React 组件。
- 不要把 CSS transition 和 Pixi ticker 混合成两套动画系统。
- 不要在业务页面组件里直接写大量场景逻辑。
- 所有场景更新节奏都应该进入统一 loop。

## 7. 目录规划

后续摩天大楼模块建议至少拆成下面几块：

```text
reception/

    visualization/
      skyscraper/
        page.tsx
      page.tsx
      layout.tsx
  components/
    skyscraper/
      skyscraper-shell.tsx
      skyscraper-canvas.tsx
      skyscraper-overlay.tsx
      skyscraper-loading.tsx
  lib/
    skyscraper/
      engine/
      scene/
      model/
      store/
      runtime/
      mappers/
  public/
    skyscraper-assets/
      modules/
      backgrounds/
      foreground/
      characters/
      fx/
      ui/

```

原则：

- 页面代码和像素资源必须分开。
- 运行时代码和资源映射逻辑必须分开。
- 静态模块统一放在 `reception/public/skyscraper-assets` 下，不要散落在组件目录。

## 8. 像素资源目录规则

`reception/public/skyscraper-assets` 是摩天大楼页面运行时静态资源的唯一入口目录。

各目录职责：

- `modules/`：楼体模块、窗户模块、屋顶模块、管道模块、霓虹标识模块
- `backgrounds/`：天空、远景楼群、远景地面、云层等背景贴图
- `foreground/`：电线、栏杆、树木、标识牌、前景遮挡层
- `characters/`：角色 sprite sheet、角色状态帧
- `fx/`：蒸汽、警报灯、粒子、扫描线、光点等特效贴图
- `ui/`：像素风场景内局部 UI 资源
- `atlases/`：打包后的 atlas 及对应描述文件
- `meta/`：调色板、尺寸规范、资源映射配置

命名建议：

- `building_core_a.png`
- `building_window_warning_on.png`
- `bg_sky_night.png`
- `fg_power_lines_a.png`
- `fx_alarm_red_loop.png`
- `char_operator_idle_strip.png`

## 8.1 这两张参考图要求的静态文件拆分

如果最终页面要接近 `bright.png` 和 `dark.png` 这种细致程度，至少要先准备下面这些静态文件，而不是只准备“一张楼图”。

### A. 楼体结构层

- 楼体外框
- 每层横向梁柱
- 玻璃幕墙外框
- 中央电梯井 / 竖向玻璃塔
- 一层门厅和门头
- 屋顶基础平台

注意：

- 这一层必须保持透视和比例完全一致，白天版和夜晚版不能换结构。
- 结构层尽量中性，不要把白天反光和夜晚发光直接烘死在同一张图里。
- 这一层是后续所有状态切换的底座。

### B. 玻璃与反射层

- 白天玻璃高光
- 夜晚玻璃暗反射
- 电梯井玻璃高亮层
- 首层大堂玻璃反射层

注意：

- 白天和夜晚的玻璃表现差异非常大，必须独立出层。
- 玻璃反射不能和室内物件画死在一起，否则无法做状态切换。
- 反射层建议半透明叠加，不建议直接混进楼体结构层。

### C. 室内基础层

- 办公室地板和墙体
- 工位桌椅
- 会议桌
- 电脑、屏幕、柜体、绿植
- 走道和隔断
- 屋顶休闲区、停机坪、设备房

注意：

- 参考图里每层房间内容不同，所以不能只复制一层办公室到所有楼层。
- 第一阶段不一定要每层完全不同，但至少要准备 4 到 6 组可轮换的室内模块。
- 室内层需要和“有人 / 无人、亮灯 / 灭灯”分开，避免后续状态切换成本过高。

### D. 室内灯光与发光层

- 办公室暖光遮罩
- 门厅光照
- 屋顶氛围灯
- 楼体 logo 发光
- 夜间屏幕亮度层

注意：

- `dark.png` 的质感很大一部分来自室内暖光，不是来自楼体本身。
- 光照层必须可独立开关，方便根据健康态、离线态、告警态做变化。
- 白天版也要有少量室内亮区，但权重应显著低于夜间版。

### E. 屋顶与设备层

- 直升机
- 停机坪标识
- 天线 / 通信塔
- 空调机组 / 设备箱
- 屋顶花坛 / 休闲家具

注意：

- 这部分是识别度很高的 silhouette，必须单独准备。
- 如果后续要做轻微动画，例如旋翼、警示灯、天线闪点，静态层和动态层要分开。
- 不要把屋顶设备全画进同一张平面图，否则后面难做遮挡和层级。

### F. 城市背景层

- 白天远景城市
- 夜晚远景城市
- 白天云层
- 夜晚云层 / 月亮
- 远景建筑亮灯点

注意：

- `bright.png` 和 `dark.png` 的远景不是装饰，而是空间感来源。
- 远景应至少分成天空层、远景楼群层、远景氛围层，后续可以做轻微视差。
- 月亮、夜云、白云不要直接和天空底图画死为一个单层大图。

### G. 街道与地面层

- 路口和斑马线
- 街边树木
- 路灯
- 停车位 / 车道线
- 地面行人剪影或占位角色
- 汽车基础形态

注意：

- 页面如果只渲染楼，不渲染底部街区，会立刻失去参考图里的“城市真实感”。
- 地面元素可以先静态，后续再把车和人替换成 sprite。
- 树和路灯属于前中景连接层，位置必须稳定，不能随意漂。

### H. 前景遮挡与氛围层

- 夜间近景树冠遮挡
- 轻雾 / 空气透视
- 夜间街灯光斑
- 少量体积云边缘

注意：

- `dark.png` 的“细腻”很大一部分来自近景遮挡和氛围压暗。
- 这些层要少而准，不能把画面糊掉。
- 氛围层应该点到为止，避免把像素风做成厚重插画滤镜。

## 8.2 页面做细致之前必须先准备的静态文件

从工程视角，真正应该提前准备的不只是 `.png`，而是一整套可维护的静态文件。

### 运行时文件

- 分层导出的 `png/webp` 贴图
- atlas 贴图集
- atlas 对应的 `json` 描述文件
- 层级顺序和 anchor 配置文件
- 场景命中区域配置文件
- 楼层到业务模块的映射配置文件

### 美术源文件

- `aseprite/psd` 分层源文件
- 白天 / 夜晚调色板文件
- 屋顶、楼层、街道的模块草图
- 导出规范说明

### 设计规范文件

- 层级命名规则
- 图层导出尺寸规则
- 逻辑分辨率规则
- 白天 / 夜晚共享结构约束
- 哪些层允许做程序变色，哪些不允许

注意：

- 如果没有源文件和导出规范，后续每次改楼体都会反复返工。
- 如果没有 atlas 和配置文件，场景一复杂就会出现资源难管理、命中错误和层级混乱。
- 如果没有白天 / 夜晚共享结构约束，两套图会越画越像两个不同项目。

## 8.3 基于 bright/dark 的准备优先级

优先级应该这样排，而不是一上来先做人物动画：

1. 楼体结构层
2. 玻璃反射层
3. 室内基础层
4. 夜间暖光 / 白天反光层
5. 屋顶设备层
6. 远景城市与天空层
7. 街道和门厅层
8. 前景遮挡层
9. 角色和车辆 sprite

原因：

- 这两张参考图的第一视觉中心永远是楼体和灯光，不是人物。
- 只要前 6 项做对，页面就已经会有“细致的游戏场景感”。
- 人物和车流是加分项，不是最先决定画面成立与否的项。

## 9. 画质与资源规范

是否“细腻”主要取决于资源方案，不是是否用了高级框架。

必须遵守的规则：

- 采用固定逻辑分辨率，例如首版使用 `640x360` 内部分辨率。
- 所有展示缩放必须使用整数倍缩放。
- 使用 nearest-neighbor，禁止平滑缩放。
- 场景坐标尽量对齐整数像素。
- 保持统一 tile 网格，首版建议以 `16x16` 为基础单位。
- 静态细节优先做进贴图，不要把所有细节都交给运行时滤镜。
- 远景、中景、主楼体、灯光、角色、特效、前景遮挡必须分层。
- `bright.png` 和 `dark.png` 这种 1024x1536 竖构图更适合作为概念母版，不适合作为最终交互场景单张贴图。
- 最终运行时应该使用“同结构、多层导出”的资源方案，而不是直接切一整张大图。

建议的细节来源：

- 静态楼体模块
- 窗户亮灭状态贴图
- 阴影和高光贴图
- 霓虹边框和标识贴图
- 前景遮挡物
- 少量循环特效贴图
- 白天 / 夜晚两套独立玻璃表现
- 门厅、屋顶、街道等高识别区域的专属模块

不建议：

- 拿照片纹理直接像素化后贴上去
- 混用不同分辨率素材
- 用 CSS 边框和圆角去模拟整栋像素建筑
- 依赖强 blur / 发光滤镜来“伪造质感”
- 直接把 `bright.png` 或 `dark.png` 当成最终可交互场景背景

## 11. 开发阶段

### Phase 0：对齐与铺底

- 固定这份方案为摩天大楼像素风开发基线
- 以 `bright.png` / `dark.png` 为母版完成楼体拆层说明
- 建立 `skyscraper-assets` 独立目录
- 明确 React / Pixi 边界
- 明确资源命名、尺寸和分层规则

验收标准：

- 白天 / 夜晚共用结构和独立表现的边界被写清楚
- 已列出首批必须导出的静态文件
- 不再把“整张概念图直接上页面”当成可行方案

### Phase 1：Pixi 场景壳层

- [x] 新建 Pixi 客户端场景组件
- [x] 完成 Pixi Application 生命周期接入
- [x] 完成内部逻辑分辨率、缩放和 resize 策略
- [x] 加入基础背景、主楼体占位资源、街道角色和基础动态布局

验收标准：

- 页面可正常渲染像素场景
- 不模糊，不抖动，不出现明显半像素偏移
- 现有页面布局不被破坏

### Phase 2：楼体静态重建

- [x] 用模块化贴图替换当前 DOM 楼体
- [ ] 将现有业务指标映射为楼体高度、灯光、状态色和告警层
- [ ] 完成楼体点击命中和跳转

验收标准：

- 主要楼体全部由 Pixi 场景渲染
- 当前可视视图下的摩天大楼页面原有业务含义保留
- 视觉上已经脱离“现代 dashboard 卡片”形态

## 12. 数据流建议

推荐数据流：

`Backend API / Realtime -> React Query / WS Layer -> Skyscraper Mapper -> Zustand Scene Store -> Pixi Engine -> Render`

解释：

- 后端仍然提供业务数据
- React 层负责接收与归一化
- `mapper` 负责把业务指标转换成楼体状态、窗户数量、动画标记和特效标记
- Pixi 只消费场景模型，不直接理解业务接口细节

## 13. 不要做的事情

- 不要为了像素风重写 FastAPI 后端
- 不要继续扩张当前页面里的 DOM 楼体实现
- 不要把贴图文件直接塞进 `app/visualization/skyscraper` 目录
- 不要用多套动画系统同时驱动同一批视觉元素
- 不要先做人物大系统再补楼体基础资源
- 不要一开始就追求 100% 完整美术，先把架构和资源管线跑通

## 14. 当前结论

对于 `XXL` 的摩天大楼页面，正确方向不是“继续优化纯 CSS 像素风”，而是：

- 以 `PixiJS` 作为主场景渲染层
- 以 `React` 作为业务壳层
- 以独立静态资源目录管理像素模块
- 先建立模块化贴图系统，再逐步补人物、日夜和环境动画

这条路线能兼顾三件事：

- 画面细节
- 工程结构
- 与现有 `XXL` 项目的兼容性
